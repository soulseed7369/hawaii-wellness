🚨Critical Blockers

These issues must be resolved before other SEO work will have meaningful impact.

⚙️
SPA Client-Side Rendering — AI Crawlers See Empty HTML
React Vite SPA with no SSR. GPTBot, ClaudeBot, PerplexityBot execute zero JavaScript — they receive an empty <div id="root"></div>. Googlebot renders JS but wastes crawl budget.
🔴 Critical Impact Hard Effort src/App.tsx index.html
▼

Why It's Blocking Everything

When GPTBot or PerplexityBot fetches https://hawaiiwellness.net/profile/123, they receive a near-empty HTML shell. No practitioner name, no modalities, no bio, no schema — nothing. As AI search grows (ChatGPT: 800M+ weekly users, Perplexity: 780M queries/month), this is costing you significant organic reach.

Verify the Problem

# Test what GPTBot sees — should return empty root div curl -H "User-Agent: Mozilla/5.0 (compatible; GPTBot/1.0; +https://openai.com/gptbot)" \ https://hawaiiwellness.net/profile/[any-id] # You'll see: <div id="root"></div>

Fix Options (Choose One)

  • Option A — Express API Layer (2-3 days): Build /api/profile/:id that fetches Supabase + renders React to static HTML using ReactDOMServer.renderToStaticMarkup. Add as Vercel serverless function.
  • Option B — Next.js Migration (2-3 weeks): Migrate from Vite to Next.js with ISR. Gold standard. Re-renders profiles every 24h. Works natively on Vercel.
  • Option C — Static Snapshots (2-3 days, hacky): Nightly cron generates static HTML for top 200 profiles. Serve via Vercel edge. Quick but brittle.

→ Recommendation: Start with Option A (API layer) this sprint; plan Next.js migration for Sprint 2.

Fix: Dynamic Sitemaps via Vercel API Routes

// src/pages/api/sitemap-practitioners.xml.ts (Vercel serverless) export default async function handler(req, res) { const { data } = await supabase .from('practitioners') .select('id, updated_at') .eq('status', 'published'); const urls = data.map(p => ` <url> <loc>https://hawaiiwellness.net/profile/${p.id}</loc> <lastmod>${p.updated_at.substring(0, 10)}</lastmod> <changefreq>weekly</changefreq> <priority>0.8</priority> </url>`).join(''); res.setHeader('Content-Type', 'application/xml'); res.setHeader('Cache-Control', 'public, max-age=86400'); // cache 24h res.send(`<?xml version="1.0"?><urlset ...>${urls}</urlset>`); }

Add links to dynamic sitemaps in public/robots.txt and main public/sitemap.xml.

Fix: Move Static Schemas to index.html

For static pages (home, about, list-your-practice), move schema directly into index.html:

<!-- index.html <head> — static schemas visible to all crawlers --> <script type="application/ld+json"> { "@context": "https://schema.org", "@type": "WebSite", "url": "https://hawaiiwellness.net", "name": "Hawaiʻi Wellness", "potentialAction": { "@type": "SearchAction", "target": "https://hawaiiwellness.net/directory?query={search_term_string}", "query-input": "required name=search_term_string" } } </script>

For dynamic pages (/profile/:id, /articles/:slug), these require SSR or the API layer to inject schema into the HTML response.

Missing Schema Types to Add

  • MedicalBusiness — use instead of generic LocalBusiness for practitioners
  • HealthAndBeautyBusiness — for wellness centers
  • FAQPage — for /help accordion content
  • WebSite with SearchAction — enables Google Sitelinks search box

🔒Privacy-Safe SEO Architecture

How to be fully crawlable by AI and Google while keeping provider phone numbers and emails protected from scrapers — with zero cloaking risk.

💡
The core principle: Contact info should never be in any HTML source — not for crawlers, not for users. It loads only when a real user clicks a button. This is the same pattern used by Airbnb, Yelp, and LinkedIn. It's universally accepted, GDPR-friendly, and eliminates any cloaking concern entirely.
🏗️ Two-Layer Architecture
LAYER 1 SSR / Crawlable HTML

Served to: Googlebot, GPTBot, ClaudeBot, PerplexityBot, social scrapers (Slack, iMessage)

✅ Name, bio, modalities
✅ City, island, geo coordinates
✅ Website URL, booking URL
✅ Schema markup (no PII fields)
✅ og:title, og:description, og:image
❌ Phone number — never here
❌ Email address — never here
LAYER 2 Click-to-Reveal (JS only)

Served to: real users after explicit button click → API call → return value

✅ Everything from Layer 1
✅ "Show Phone" button → fetches live
✅ "Show Email" button → fetches live
✅ RequestInfoModal (existing)
✅ Rate-limited, logged per view
✅ GDPR-compliant access log
✅ Why This Has Zero Cloaking Risk

Google's cloaking policy prohibits showing crawlers different content than users see. The click-to-reveal pattern is not cloaking because:

  • The HTML source is identical for crawlers and users — both receive the same server-rendered HTML
  • Contact info is absent from the HTML source for everyone, not just crawlers
  • It loads via a separate API call triggered by a click — the same way a user would interact with it
  • This is explicitly the pattern Google documents as acceptable for gated/progressive content
  • Airbnb, Yelp, LinkedIn, Booking.com all use this exact pattern — no SEO penalties

The previous two-layer idea (serve different HTML to bots vs. users) carried cloaking risk. This approach has none.
